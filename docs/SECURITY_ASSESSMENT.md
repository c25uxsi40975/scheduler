# セキュリティ評価レポート

**対象アプリケーション**: 外勤調整システム (Streamlit + Google Sheets)
**評価日**: 2026-02-28
**評価範囲**: アプリケーション全体（認証・データストア・通信・UI）

---

## 概要

本アプリケーションはStreamlitベースのWebアプリであり、Googleスプレッドシートをデータベースとして使用している。Google Sheets APIを介した認証にはGCPサービスアカウントを利用し、医員・管理者の認証にはスプレッドシート上に保存されたパスワードハッシュを使用している。

以下に、重大度別に脆弱性と推奨対策をまとめる。

---

## 重大度: Critical（致命的）

### 1. パスワードハッシュの脆弱性（ソルトなしSHA-256）

**該当箇所**: `database/connection.py:154`

```python
def _hash_password(password: str) -> str:
    return hashlib.sha256(password.encode()).hexdigest()
```

**問題点**:
- ソルト（salt）を使用していないため、同じパスワードは常に同じハッシュ値になる
- SHA-256は高速なハッシュ関数であり、パスワードハッシュ用途には不適切（GPUによる高速ブルートフォースが可能）
- レインボーテーブル攻撃に脆弱
- 複数ユーザーが同じパスワードを使用している場合、ハッシュ値の一致から判明する

**推奨対策**:
- `bcrypt`、`scrypt`、または`argon2`などの専用パスワードハッシュ関数に移行する
- 移行例:
  ```python
  import bcrypt
  def _hash_password(password: str) -> str:
      return bcrypt.hashpw(password.encode(), bcrypt.gensalt()).decode()
  def _verify_password(password: str, hashed: str) -> bool:
      return bcrypt.checkpw(password.encode(), hashed.encode())
  ```

### 2. パスワードハッシュがスプレッドシート上に平文保存

**該当箇所**: `database/connection.py:161`（医員マスタのスキーマ）、`database/master.py:52`

**問題点**:
- `password_hash`カラムがスプレッドシート上に直接保存されている
- スプレッドシートの共有設定や共同編集者を通じて、全ユーザーのパスワードハッシュが閲覧可能
- ソルトなしSHA-256のため、ハッシュ値からパスワードの逆引きが容易
- Googleスプレッドシートの変更履歴からも過去のハッシュ値を取得可能

**推奨対策**:
- パスワード認証を外部の認証サービス（Firebase Auth、Auth0等）に委譲する
- スプレッドシートへのアクセス権限を最小限に制限する（サービスアカウントのみ）
- 管理者パスワードも`設定`シートに保存されている（`database/auth.py:51`）ため同様の対策が必要

### 3. デフォルトパスワード「1111」の自動設定

**該当箇所**: `database/connection.py:228-241`、`database/master.py:63`

**問題点**:
- 全医員のデフォルトパスワードが`"1111"`で固定されている
- アプリ起動時（`init_db()`）にパスワード未設定の全医員に自動適用
- 新規医員追加時もデフォルト値が`"1111"`（`admin_master.py:89`）
- パスワード変更を強制する仕組みがない
- 攻撃者はアカウント名を推測し`"1111"`でログインを試行できる

**推奨対策**:
- デフォルトパスワードの使用を廃止し、初回ログイン時にパスワード設定を強制する
- パスワードの複雑性要件を追加する（最低文字数、英数混合等）
- 管理者が初期パスワードを個別に設定できるUIを（現在はあるが）デフォルト値を除去する

## 重大度: High（高）

### 4. ログインに対するレート制限・アカウントロックの欠如

**該当箇所**: `app.py:98-103`（管理者ログイン）、`app.py:119-131`（医員ログイン）

**問題点**:
- ログイン試行回数に制限がない
- ブルートフォース攻撃やパスワードスプレー攻撃に対する防御がない
- デフォルトパスワード`"1111"`と組み合わせると、アカウント乗っ取りが極めて容易

**推奨対策**:
- ログイン試行の回数制限を実装（例: 5回失敗で一定時間ロック）
- CAPTCHA導入
- ログイン失敗ログの記録（監査証跡）

### 5. セッション管理の脆弱性

**該当箇所**: `app.py:54-61`

**問題点**:
- セッションタイムアウトが設定されていない（`st.session_state`はブラウザタブが開いている限り有効）
- 認証済みセッションの無期限維持
- ログアウト時のセッション無効化が不完全（`st.session_state`の値をクリアするのみで、サーバー側でのトークン無効化なし）
- CSRFトークンの実装なし（Streamlitのデフォルト機能に依存）

**推奨対策**:
- セッションタイムアウトの実装（例: 30分で自動ログアウト）
- 認証状態にタイムスタンプを含め、定期的に再認証を要求

### 6. スプレッドシートへの無制限アクセス（サービスアカウント権限）

**該当箇所**: `database/connection.py:38-58`

**問題点**:
- 単一のサービスアカウントが2つのスプレッドシート全体に対してフルアクセス権を持つ
- 行レベル・列レベルのアクセス制御が不可能（Google Sheets APIの制限）
- サービスアカウントキーが漏洩した場合、全データ（医員情報、パスワードハッシュ、メールアドレス、スケジュール等）が流出する
- 認証されたユーザーであれば、アプリを通じて本来アクセスすべきでないデータにもアクセス可能

**推奨対策**:
- サービスアカウントキーの安全な管理（ローテーション、最小権限の原則）
- 可能であれば、適切なアクセス制御を持つデータベース（PostgreSQL等）への移行を検討
- スプレッドシートの共有設定を確認し、サービスアカウント以外の閲覧者を制限

### 7. 認可チェックの不足（水平権限昇格）

**該当箇所**: `pages/doctor_input.py`、`database/auth.py`

**問題点**:
- 医員ログイン後、`st.session_state.doctor_id`を通じてのみアクセス制御
- Streamlitの`session_state`はサーバーサイドで管理されるが、ロジック上の認可チェックが不足
- 例: `upsert_preference()`は`doctor_id`パラメータを直接受け取り、呼び出し元の認証情報との整合性チェックがない
- 管理者ページに対するロールベースアクセス制御が`session_state`のフラグのみに依存

**推奨対策**:
- データ操作関数にアクセス制御チェックを追加
- 現在のセッションユーザーとリクエストされたリソースの所有者を照合

## 重大度: Medium（中）

### 8. XSS（クロスサイトスクリプティング）のリスク

**該当箇所**: `app.py:33`、`admin_master.py:78`、`admin_master.py:136`、`admin_master.py:316`

```python
st.markdown(..., unsafe_allow_html=True)
```

**問題点**:
- `unsafe_allow_html=True`が複数箇所で使用されている
- ユーザー入力（医員名、外勤先名等）がサニタイズされずにHTML内に埋め込まれる可能性
- `admin_master.py:137`では医員名がマークダウン内に直接挿入される:
  ```python
  st.markdown(f"**{d['name']}**　{status_label}　ID: {id_display}...")
  ```
- 医員名に悪意のあるHTMLやJavaScriptが含まれている場合、XSSが成立する可能性がある

**推奨対策**:
- `unsafe_allow_html=True`の使用を最小限に制限
- ユーザー入力をHTML出力前にエスケープ処理する
- CSSの適用にはStreamlitのテーマ機能やカスタムコンポーネントを使用する

### 9. GAS Web Appエンドポイントの認証不足

**該当箇所**: `gas/reminder.gs:56-75`、`pages/doctor_input.py:33-57`、`pages/admin_generate.py:133-140`

**問題点**:
- GAS Web Appの`doPost()`にはリクエスト元の認証・認可チェックがない
- エンドポイントURL（`gas_webapp_url`）を知る者は誰でもリクエストを送信可能
- 不正なリクエストによりスパムメールの送信が可能
- `action`パラメータを操作して意図しないメール送信をトリガーできる
- SSRF（Server-Side Request Forgery）のリスク: `gas_webapp_url`が改竄された場合、内部ネットワークへのリクエストが発生する可能性

**推奨対策**:
- GAS Web AppにAPIキーまたはシークレットトークンによる認証を追加
- リクエスト元IPの制限またはGoogleサービスアカウントによる認証
- `gas_webapp_url`の値を検証（ホワイトリスト方式）

### 10. 入力値のバリデーション不足

**該当箇所**: 各CRUD操作関数全般

**問題点**:
- 医員名、外勤先名、メールアドレス等の入力値に対する検証が最小限
- メールアドレスのフォーマット検証なし（`database/auth.py:138-148`）
- 数値フィールドの範囲チェックが一部欠如
- JSON形式で保存されるフィールド（`preferred_doctors`、`fixed_doctors`等）にインジェクションの余地がある可能性

**推奨対策**:
- メールアドレスのフォーマット検証を追加
- 全入力値に対する型・範囲・長さのバリデーション
- サーバーサイドでの入力サニタイズの徹底

### 11. エラーハンドリングによる情報漏洩

**該当箇所**: 各所

**問題点**:
- API呼び出しのエラーメッセージがそのままユーザーに表示される可能性
- スプレッドシートキーやサービスアカウント情報がスタックトレースに含まれる可能性
- `gspread.exceptions.APIError`のエラーメッセージにはAPIキー等の機密情報が含まれる場合がある

**推奨対策**:
- ユーザーに表示するエラーメッセージを汎用的なものにする
- 詳細なエラーログはサーバーサイドのログに記録
- 本番環境でのデバッグ情報の表示を無効化

### 12. 同時アクセス時のデータ競合

**該当箇所**: `database/operational.py`、`database/master.py`

**問題点**:
- Google Sheetsへの同時書き込みに対するロック機構がない
- 複数の管理者・医員が同時にデータを更新した場合、データの不整合が発生する可能性
- `_next_id()`によるID採番が競合する可能性（2つのリクエストが同じIDを取得する）
- `_find_row_index()`と`update_cell()`の間にレースコンディションが発生する可能性

**推奨対策**:
- 楽観的ロック（バージョン番号によるコンフリクト検出）の導入
- ID採番にUUIDを使用
- クリティカルな操作には排他制御を追加

## 重大度: Low（低）

### 13. HTTPSの非強制

**問題点**:
- Streamlit Community Cloudではデフォルトでhttps化されるが、ローカルやオンプレミス環境では`http`で起動される
- パスワードを含むフォーム送信が暗号化されない可能性

**推奨対策**:
- 本番環境ではリバースプロキシ（nginx等）でHTTPS終端を行う
- Streamlitの設定でHTTPS証明書を指定

### 14. IDの連番による列挙可能性

**該当箇所**: `database/connection.py:145-151`

**問題点**:
- 医員ID、外勤先IDが連番で採番されるため、有効なIDの推測・列挙が容易
- APIエンドポイントを直接操作する攻撃には限定的な影響だが、内部ロジックの理解を容易にする

**推奨対策**:
- UUIDの使用を検討（ただし、Google Sheets上での管理が複雑になるトレードオフあり）

### 15. パスワード変更時の既存セッション維持

**該当箇所**: `app.py:238-252`

**問題点**:
- 医員がパスワードを変更しても、他のブラウザ/デバイスの既存セッションは無効化されない
- パスワード漏洩時のアカウント回復後も攻撃者のセッションが有効なまま残る可能性

**推奨対策**:
- パスワード変更時に全セッションを無効化する仕組みを追加

### 16. 機密データのキャッシュ

**該当箇所**: `database/master.py:42-43`、`database/auth.py:17-18`

```python
@st.cache_data(ttl=120)
def get_doctors(active_only=True):
```

**問題点**:
- `st.cache_data`によりパスワードハッシュを含む医員データがStreamlitのサーバーメモリにキャッシュされる
- TTL（120秒）の間、古いデータが返される可能性がある（パスワード変更直後に旧パスワードで認証成功するリスク）
- メモリダンプ等でキャッシュからハッシュ値が漏洩する可能性

**推奨対策**:
- パスワードハッシュをキャッシュ対象から除外
- 認証関連のデータは毎回スプレッドシートから取得

## Google Sheets特有のリスク

### A. データベースとしてのGoogle Sheetsの根本的制限

| 観点 | リスク |
|------|--------|
| アクセス制御 | 行・列レベルのアクセス制御不可。シート全体の共有/非共有のみ |
| 監査証跡 | アプリ経由の変更とスプレッドシート直接編集の区別が不可能 |
| トランザクション | ACID特性なし。部分的な更新が発生する可能性 |
| データ容量 | 1シート最大1000万セル、5MB制限。大規模運用に不適 |
| API制限 | Google Sheets APIにはレート制限あり（100リクエスト/100秒/ユーザー） |
| バックアップ | 自動バックアップ機能なし（Google Driveのバージョン履歴に依存） |
| 暗号化 | アプリケーションレベルの暗号化なし（Google側のサーバー暗号化のみ） |
| データ分離 | マスタデータと運用データを2つのスプレッドシートに分離しているが、同一サービスアカウントでアクセス |

### B. スプレッドシート直接編集によるデータ改竄

**問題点**:
- スプレッドシートにアクセス権を持つユーザーは、Google Sheets UIから直接データを編集可能
- パスワードハッシュの書き換え、is_activeフラグの操作、スケジュールの改竄が可能
- アプリのバリデーションを完全にバイパスできる
- 変更履歴は残るが、改竄検知の仕組みがない

**推奨対策**:
- スプレッドシートの共有設定をサービスアカウントのみの「編集者」に制限
- 人間ユーザーには「閲覧者」権限のみ付与
- GASトリガーによる変更監視・通知の導入を検討

## 総合評価

本アプリケーションは**内部利用を想定した業務ツール**であり、インターネット上で広く公開される前提ではないと思われるが、以下の点で改善が必要である:

1. **認証基盤が脆弱**: ソルトなしSHA-256 + デフォルトパスワード + レート制限なしの組み合わせは、最小限の攻撃スキルでもアカウント乗っ取りが可能
2. **Google Sheetsをデータベースとして使用する設計上の制限**: アクセス制御・トランザクション・監査の観点で根本的な制約がある
3. **機密データの露出**: パスワードハッシュ・メールアドレスが適切に保護されていない

短期的には優先度1-5の対策を、中長期的にはデータベース移行を含めたアーキテクチャの見直しを推奨する。
